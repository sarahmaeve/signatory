# Signatory: v0.1 Invariants

## Status

Accepted — 2026-04-17. Binding scope document for v0.1. Supersedes

## Motivation

V0.1 is dogfood-scoped: a solo developer with an LLM pair runs it
locally against the dependencies they're considering. Between the
original scoping and now, two constraints sharpened:

1. **Token cost is higher than the original plan anticipated.** Any
   design that routes mechanical enumeration through an agent is
   paying a premium for work that a Go program does better, more
   cheaply, and reproducibly.
2. **Agent-to-agent communication is less capable than the original
   plan assumed.** Subagents don't share memory. Prose summaries are
   lossy. `WebFetch` and `WebSearch` are not durable data transports.

These constraints do not change what signatory *is* (a supply-chain
trust analysis tool whose product is signal + analysis). They change
what we build *first* and, critically, what we refuse to build in
v0.1 even if it looks cheap.

This document fixes four invariants. Each is stated, justified,
and made enforceable — either by a test, a lint, or a reviewable
architectural rule. Violations of an invariant are a v0.1 blocker,
not a footnote.

## Invariant 1: No direct Anthropic API access

**Statement.** Signatory's Go code does not call the Anthropic API,
does not hold or read an Anthropic API key, and does not import any
Anthropic SDK. All LLM work in v0.1 is dispatched by the Claude
Code orchestrator as subagents, which inherit the operator's Max
subscription.

**Rationale.**
- The v0.1 deployment shape is a single-user local CLI plus MCP
  server. There is no place for a hosted API call to land; anything
  that needs one is out of scope.
- Max-subscription dispatch bounds cost by session activity instead
  of per-token billing. This aligns with the dogfood target user.
- Eliminating the API path removes an entire class of v0.1
  concerns: key management, rate limiting, billing surface, secret
  rotation, abuse controls, and the subtle pressure those create
  toward "just a small daemon that does enrichment in the
  background."

**What is still allowed.**
- The orchestrator (a human operator or Claude Code session)
  invoking `Agent` tool calls to dispatch subagents. That's the
  entire LLM surface.
- Subagents using `WebFetch` / `WebSearch` against third-party
  sources (GitHub, registries, advisories). Those are data
  collection, not LLM calls.
- Future v0.2+ designs that re-introduce an API path. They are
  out of scope here, not forbidden forever.

**Enforcement.**
1. A build-time grep (CI step or pre-commit hook) that fails on any
   import matching `github.com/anthropics/anthropic-sdk-go`,
   `github.com/sashabaranov/go-openai`, or equivalent LLM-client
   packages. Single-line check; easy to extend.
2. A unit test in `internal/` that asserts the module graph contains
   no such dependency. `go list -m all | grep` is sufficient.
3. A documented review rule: any PR that introduces an
   `ANTHROPIC_*`, `OPENAI_*`, or `LLM_*` env var read is rejected
   in v0.1 without a scope-change approval.

**Explicit non-goals covered by this invariant.**
- Background enrichment daemons.
- Scheduled re-analysis jobs that dispatch LLM calls without a
  user session.
- Hosted mode (SaaS, multi-tenant). Everything that implies a
  server-side LLM caller is v0.2+.

## Invariant 2: Mechanical collection is Go, not agents

**Statement.** Every signal that can be extracted by a deterministic
program is extracted by a deterministic program. Analyst subagents
do not enumerate, list, count, or grep as their primary task; they
reason over a pre-assembled signal bundle, and only read source code
to verify specific hypotheses they are forming.

**Rationale.**
- The atuin worked example (`design/mcp-dual-analyst-architecture.md`
  §"Reference") showed that most analyst conclusions had a collector
  component: the analyst's judgment layer earned its cost on
  composition and reasoning, not on enumeration. Enumeration routed
  through an agent is expensive, non-deterministic, and incomplete.
- Deterministic collectors are reproducible. Two runs produce the
  same signals on the same target commit. Analyst conclusions
  downstream of reproducible signals are themselves more stable.
- Human-launched collection is a feature, not a limitation. A human
  running `signatory collect <target>` before inviting an analyst
  is explicit consent to the cost and time of data gathering.

**What this means concretely.**
- `signatory collect <target>` is a single CLI entry point that
  runs every collector applicable to the target and writes signals
  to SQLite. It exits with a short stdout summary, not a flood.
- Analyst subagents in `/analyze` receive the signal bundle via the
  MCP read path. They do not run `gh api`, `git log`, or `ls` as
  bulk enumeration. They may `Read` specific files whose paths are
  named in a signal, as verification.
- Collectors live under `internal/signal/{git,source,registry/npm,…}/`.
  Each has table-driven tests. None imports an LLM client
  (Invariant 1).

**Layer 1 collector scope for v0.1.**
See `mcp-dual-analyst-architecture.md` §"Layer 1" for the full
list. v0.1 ships the following:

| Collector | Directory | Ships in v0.1 |
|-----------|-----------|---------------|
| GitHub API | `internal/signal/github/` | ✓ (extend existing) |
| OpenSSF Scorecard | `internal/signal/scorecard/` | ✓ (extend existing) |
| git (local clone) | `internal/signal/git/` | ✓ (new) |
| source grep/AST | `internal/signal/source/` | ✓ (new, minimal set) |
| npm registry | `internal/signal/registry/npm/` | ✓ (new) |
| PyPI registry | `internal/signal/registry/pypi/` | defer to v0.2 unless trivial |

The "minimal set" for source grep in v0.1: hardcoded URL
extraction, listener discovery, file-permission-call listing,
temp-file-path generation pattern detection. These are the four
patterns the atuin analysis proved load-bearing. Everything else
waits until a real target demands it.

**Enforcement.**
1. Analyst subagent allowlists in `.claude/skills/analyze/SKILL.md`
   and handoff templates must not include `Bash`. This already
   holds and is a static check: grep the skill and handoff files.
2. `signatory collect <target>` must populate at least the Layer 1
   signals the analyst handoff references. Integration test: run
   `signatory collect` against a fixture target, then assert the
   analyst's handoff signal bundle is non-empty.
3. Review rule: any new capability added to the `/analyze` skill
   that asks an analyst to enumerate data (rather than interpret
   pre-collected data) is rejected without a collector-first
   alternative proposal.

**Exception.** If a signal requires judgment inherently —
"does this CI workflow composition gate releases behind review?" —
it stays in Layer 2. The invariant is *no mechanical work in
agents*, not *no agents*. Judgment is what agents are for.

## Invariant 3: SQLite is the canonical store; scratch files are not

**Statement.** The signatory SQLite database is the only
authoritative source of signals, analyses, conclusions, and
postures. Markdown and JSON files under `filestore/analysis/` and
`design/{analysis,dogfood}/` are either (a) working-data artifacts
that a collector needs (e.g., a git clone), or (b) rendered
snapshots of store contents. They are never the source of truth,
and nothing reads them as a substitute for a store query.

**Rationale.**
- When two stores coexist, they drift. The current design has
  `.md` files checked into `design/analysis/` that were produced by
  the old analysis workflow, plus a SQLite store that the new
  workflow populates. A reader today cannot tell, without
  archaeology, whether a file or the database is current.
- The MCP surface *already* makes this choice: the tools operate
  on the store, not on files. Every file that exists outside the
  store is dark matter to any LLM agent that reads signatory state
  via MCP.
- Intermediate scratch files across agent handoffs (`.md` → `.json`
  → ingest) are fragile. If any step fails after the subagent
  exits, the operator is left reconstructing state from transcript.

**What this means concretely.**
- Analyst subagents write their output via an MCP write tool
  (`signatory_ingest_analysis`), not via `Write` to
  `filestore/analysis/*.md`. Payload is v1-schema JSON.
- The `signatory build-output` markdown→JSON converter is
  deprecated for live pipeline use. It remains useful as an offline
  import tool for legacy markdown (there is some) but is not in
  the `/analyze` critical path.
- Human-readable outputs are rendered on demand: `signatory show
  analysis <target>` produces markdown from the store. The
  rendered markdown is a derived artifact, not a source.
- The pipeline session service (`signatory serve`) backs its
  session state with SQLite rather than in-memory maps, so a
  restart mid-run does not lose the handoff and partial outputs.
- `filestore/clones/` stays. Git requires a real filesystem;
  the clone is working data for a collector run, not scratch-pad
  communication between agents.
- `design/analysis/*.md` and `design/dogfood/*.md`:
  - Historic documents remain for reference, marked as such.
  - New assessments are ingested into the store and rendered when
    wanted. No new hand-authored analysis files are added under
    these paths as primary records.

**Enforcement.**
1. The `/analyze` skill is rewritten to dispatch analysts with
   `mcp__signatory__signatory_ingest_analysis` in their allowlist
   and to remove the `Write` + `build-output` + `ingest` dance
   from Step 3. Skill diff is the unit of review.
2. A contract test on the analyst handoff: the handoff text must
   instruct the analyst to call the MCP ingest tool, and must not
   instruct it to write to `filestore/`.
3. A schema-level test: `filestore/analysis/` contains only files
   produced by `signatory show` (render), or is gitignored entirely.
4. Migration note: existing `design/analysis/*.md` and
   `design/dogfood/*.md` get one-shot ingested into the store
   (best-effort, via `build-output` + `ingest`), then the files
   stay as human-readable historical records. New work does not
   add to these directories.

**Exception.** The pipeline service may keep a short-lived
in-memory cache in front of SQLite for latency. The canonical
record is still SQLite; the cache is a cache.

## Invariant 4: Agent communication uses named transports with named payloads

**Statement.** Communication between the orchestrator, subagents,
the store, and external sources uses only the transports and
payload formats in the table below. No other transport is used in
v0.1 for data flow; "just have the agent tell me in its reply" is
not a transport.

| Direction | Transport | Payload format |
|-----------|-----------|----------------|
| Orchestrator → subagent (task) | `Agent` tool prompt | Short natural-language prompt + a WebFetch URL |
| Orchestrator → subagent (instructions) | `WebFetch` to `127.0.0.1:21517` | Plain-text handoff from `signatory handoff` |
| Subagent → store (output) | MCP write tool (`signatory_ingest_analysis`) | v1-schema JSON |
| Subagent → store (read) | MCP read tools (`signatory_*`) | v1-schema JSON |
| Subagent → subagent (shared context) | Via store (write, then next agent reads) | v1-schema JSON |
| Subagent → external world | `WebFetch` / `WebSearch` | whatever the source returns |
| Orchestrator → user (final product) | `signatory show *` rendered from store | Markdown / terminal output |

**Rationale.**
- Subagents do not share memory. They cannot be assumed to know
  anything not in their prompt or reachable via their tools.
- Parent-to-child prose is a valid instruction channel but a poor
  data channel: the parent has to fit the data into a prompt, and
  the child has to interpret it, losing structure on both sides.
- Child-to-parent prose (the `Agent` tool's return value) is a
  valid *summary* channel — the parent reads it for sanity-check
  and progress — but not a data channel. Structured output from a
  child must land in the store, where any subsequent agent can
  query it with the same guarantees.
- Using `WebSearch` as an internal transport (one agent puts data
  on the web so another can search for it) is a non-starter:
  indexing lag, retrieval nondeterminism, privacy leakage. Don't.

**What this means concretely.**
- The pipeline service on `127.0.0.1:21517` already exists. It is
  the single authorized way to deliver handoff instructions into a
  subagent. No file-drop substitute.
- Subagents must have the relevant `mcp__signatory__*` tools in
  their allowlists — the read set always, the write set when
  they produce store-bound output.
- When the `/analyze` skill needs a second analyst to build on a
  first analyst's output, the first analyst writes to the store
  via MCP ingest; the second analyst reads via MCP query. There
  is no "have the orchestrator pass a blob of markdown from A to
  B" path.
- Synthesis works the same way: the synthesist reads analyses
  and conclusions from the store, then writes its narrative
  output back via a named ingest path. The final narrative is a
  store record with a render, not an orphan file.

**Enforcement.**
1. Subagent allowlist audits in `/analyze` and related skills:
   each subagent's declared tools must match one of the rows in
   the transport table. A subagent with `Write` and no MCP write
   tool is a bug.
2. Handoff templates (`signatory handoff`) must instruct analysts
   to use the named transports. Snapshot tests of generated
   handoffs assert the relevant phrases.
3. A pipeline integration test dispatches a stub subagent that
   reports a fixed v1-schema JSON payload via the MCP ingest
   tool; the test asserts the store ends up with the expected
   conclusion, and that `filestore/analysis/` is untouched.

**Exception.** During development, an operator may hand-run
`signatory ingest file.json` to land a legacy markdown-derived
analysis into the store. That is a one-shot migration tool,
not a pipeline transport.

## What "v0.1 done" means, concretely

A v0.1 release is acceptable when all four invariants hold and the
following observable behaviors are true:

1. `signatory collect <go-module-target>` populates a usable signal
   bundle for a real Go module (e.g., `stretchr/testify`) into
   SQLite, end to end, with no LLM involvement.
2. `signatory collect <npm-package-target>` does the same for a
   real npm package.
3. `/analyze <target>` in Claude Code:
   - Checks the store first (Invariant 3);
   - Runs `signatory collect` if needed (Invariant 2);
   - Dispatches analyst subagents with no `Bash` and with
     `mcp__signatory__*` tools, reading from and writing to the
     store via named transports (Invariants 3 and 4);
   - Produces a synthesized narrative readable via
     `signatory show analysis <target>`, rendered from the store.
4. The signatory binary does not import an LLM client package
   (Invariant 1), verified by a CI step.
5. `filestore/analysis/` is gitignored or contains only rendered
   snapshots (Invariant 3).

Each item is testable. The invariants doc and the test suite move
in lockstep: adding an invariant requires a test; removing a test
requires proposing a scope change.

## Non-goals for v0.1

Called out here so that "but it would be easy" doesn't sneak them
in:

- Hosted mode, multi-tenant mode, or any deployment that expects a
  server-side LLM caller.
- Scheduled or event-driven re-analysis. All analysis in v0.1 is
  operator-triggered.
- Federated signal sharing, burn-list subscription, cross-org
  realm identity. All deferred.
- Signing / vetted-frozen attestation envelopes. Already deferred
  to v0.3 per ROADMAP; repeated here for completeness.
- UI beyond the CLI and MCP. No web UI, no TUI, no dashboard
  rendering beyond `signatory survey`.

## Open questions

These do not block v0.1 on the invariants but want resolution
before the first public release:

1. **Legacy markdown migration.** Should the ~10
   `design/analysis/*.md` and `design/dogfood/*.md` files be
   ingested into the store via a best-effort parser, left as
   historical records, or both? Current lean: both — ingest where
   the parse succeeds cleanly, keep the file for history either
   way.
2. **Pipeline-service port and TLS.** `127.0.0.1:21517` with
   mkcert works but is a per-machine setup step. Is there a path
   to zero-install (e.g., unix socket + CA-less HTTPS that
   WebFetch tolerates)? If not in v0.1, document the setup cost
   clearly in README.
3. **MCP write tool surface.** Is `signatory_ingest_analysis`
   one tool or three (`_analysis`, `_observations`,
   `_methodology`)? Current lean: one tool with a v1-schema
   payload that already separates these; the surface stays small.
4. **Collector parallelism.** `signatory collect` could run
   collectors concurrently. v0.1 lean: run sequentially, keep it
   simple, revisit after we have a real target with enough
   collectors to make parallelism worth the flakiness budget.
5. **Rendered-markdown cache.** When `signatory show analysis`
   produces markdown from the store, should the rendered output
   be cached, or re-rendered each time? Lean: re-render; the cost
   is low and it keeps "the store is truth" honest.

## Sequencing

The invariants above imply an ordered work list. The first item
that unblocks the most is at the top.

1. **Land this document.** It becomes the reference for the rest.
2. **Add the Invariant-1 CI check.** Free guardrail, prevents
   regression during the rest of the work.
3. **Ship `internal/signal/git/`.** Removes the largest chunk of
   mechanical analyst work. See ROADMAP Layer 1 scope.
4. **Add `signatory_ingest_analysis` MCP write tool.** Enables
   Invariant 3's main rewrite.
5. **Rewrite `/analyze` skill Step 3** to use the MCP write tool
   directly from analyst subagents. Kill the markdown-to-JSON
   dance.
6. **Ship `internal/signal/source/` (minimal set).** Removes the
   rest of the analyst enumeration burden on code-level signals.
7. **Ship `internal/signal/registry/npm/`.** Unblocks the
   highest-risk ecosystem target.
8. **Back pipeline service with SQLite.** Restartability; closes
   out Invariant 3's last mismatched store.
9. **CI wire-up of the v0.1 done-checks.** Locks in the
   invariants for future regression.

Each item is small enough to land in a single session. Items 3–7
each have a matching entry in `ROADMAP.md` §"V0.1 — Must-Do"; this
document does not change that list, it orders it.
