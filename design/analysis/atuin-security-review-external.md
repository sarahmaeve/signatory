# atuin — Security Review

Skeptical security read of the `atuin` repository (shell-history recorder/replacer with optional encrypted sync and an AI subsystem). Review was performed against commit `5471119c` on `main` (version 18.14.1).

## What this thing actually is

`atuin` is a Rust workspace that does four distinct things. It's important to separate them because their risk profiles differ sharply:

1. **Shell-history capture** — sqlite-backed replacement for `~/.bash_history` etc. (crates: `atuin-client`, `atuin-history`)
2. **Optional E2E-encrypted sync** — to a server you can self-host (crates: `atuin-server`, `atuin-server-postgres`, `atuin-server-sqlite`)
3. **A local gRPC daemon** — for fast search and batched syncing (crate: `atuin-daemon`)
4. **Atuin AI** — a newer LLM-driven agent that can read files, run shell commands, and search your history under instruction from a remote server. (crate: `atuin-ai`, plus hook installers in `atuin/src/command/client/hook.rs`)

The first two are mature, well-engineered, and benign. **#3 and especially #4 are where the real security questions live.**

---

## 1. Install flow — what `curl … | sh` does

`install.sh` (and the public `setup.atuin.sh` redirect — both identical in intent) does the following *on your machine*, with your user privileges:

- Pipes `https://github.com/atuinsh/atuin/releases/latest/download/atuin-installer.sh` into `sh`. That inner script is a standard `cargo-dist` installer — no pinning, so you implicitly trust the GitHub release content for whatever "latest" is.
- Fetches `https://raw.githubusercontent.com/rcaloras/bash-preexec/master/bash-preexec.sh` (unpinned, unsigned) and drops it at `~/.bash-preexec.sh`. This is an external repo outside atuinsh's control.
- Appends `eval "$(atuin init …)"` lines to `~/.zshrc`, `~/.bashrc`, `~/.config/fish/config.fish`.
- **Silently auto-installs agent hooks into `~/.claude/settings.json`, `~/.codex/hooks.json`, `~/.pi/agent/extensions/atuin.ts`** if it detects those agents are installed (`install.sh:106-108`, implementation in `crates/atuin/src/command/client/hook.rs`). There is no separate "may I modify your AI-agent configs?" prompt — only a one-liner log.
- Interactively calls `atuin register` → signs you up with `api.atuin.sh` by default.
- Finally runs `atuin setup`, which flips `ai.enabled = true` and `daemon.enabled = true` if you press Enter (both default to "Y").

That's a lot happening in one curl-pipe. Nothing here is *malicious*, but if you're the kind of person who is uncomfortable with `curl | sh`, this is a strong case to download the binary (or build from source) and skip the script.

## 2. Encryption claim — does "fully encrypted sync" hold up?

**Yes, the claim is honest.** Both paths verified:

- Legacy history: `crates/atuin-client/src/encryption.rs` — XSalsa20Poly1305 secretbox with a 32-byte random key generated by `OsRng`, stored at `~/.local/share/atuin/key`. Serialized with base64 + msgpack.
- New record store (v2 sync, kv, dotfiles, scripts): `crates/atuin-client/src/record/encryption.rs` — PASETO v4 Local with envelope encryption (per-record CEK wrapped by a master key). Additional data (record id, host, tag, idx, version) is bound as an implicit assertion, so tampering with the metadata fails auth.

The master key never leaves the client. The server gets a bcrypt'd (`crates/atuin-server/src/handlers/user.rs`) password for auth, plus ciphertext for storage. The server genuinely cannot decrypt.

**Caveats** to be aware of:
- Single symmetric key for everything — if `~/.local/share/atuin/key` leaks, every record on every device is decryptable.
- Key rotation is not automatic. Re-keying requires `atuin store rekey` which re-encrypts all records.
- The key file's permissions aren't explicitly hardened to `0o600` on creation — it inherits umask. Only the meta DB explicitly sets `0o600` (`crates/atuin-client/src/meta.rs:69-70`).
- The `secrets_filter` regex set (`crates/atuin-client/src/secrets.rs`) catches AWS keys, GitHub PATs, Slack/Stripe/Netlify/npm/Pulumi tokens — **not** things like `DATABASE_URL=postgres://user:pw@…`, generic bearer tokens, passwords passed via `-p`, or anything behind an unknown env-var name. Treat it as best-effort, not a guarantee.

## 3. What actually gets stored and synced

For every command recorded (`crates/atuin-client/src/history.rs`): `command`, `timestamp`, `duration`, `exit`, `cwd`, `session` (uuid), `hostname:user`, `author`, optional `intent`. Encrypted before leaving the machine. No process tree, no env vars, no TTY contents — just what you typed.

## 4. Self-hosting: the asymmetric answer

| Component | Self-hostable? | Notes |
|---|---|---|
| **Sync server** (`atuin-server`) | **Yes, fully.** | Postgres 14+ or SQLite. Override `sync_address` in config, or `ATUIN_SYNC_ADDRESS`. See `docs/docs/self-hosting/server-setup.md`. |
| **Binaries** | Yes | Build from source, mirror releases, skip `install.sh`. |
| **Agent hooks, daemon, shell plugin** | Yes | All local, no network. |
| **Update-check** | **No (by default).** | `crates/atuin-client/src/api_client.rs:143-161` — `latest_version()` is **hardcoded to `https://api.atuin.sh`** regardless of your `sync_address`. Kill switch: set `update_check = false`. |
| **Atuin Hub / Atuin AI** | **No.** | `crates/atuin-ai/src/commands/inline.rs:47` falls back to `https://hub.atuin.sh`; `ai.endpoint` exists as an override, but there is no open-source Hub server in this repo. If you self-host sync, AI still wants you to authenticate at `hub.atuin.sh` — this is intentional per changelog PR [#3301](https://github.com/atuinsh/atuin/issues/3301) ("Call ensure_hub_session even if primary sync endpoint is self-hosted"). |

So: sync and history capture are fully private; **AI is not**, and the update-check phones `api.atuin.sh` even for self-hosters.

## 5. The AI crate is where I'd slow down

`crates/atuin-ai` is a relatively new subsystem (visible in the changelog: "Client-tool execution + permission system" [#3370](https://github.com/atuinsh/atuin/issues/3370), "Track coding agent shell usage" [#3388](https://github.com/atuinsh/atuin/issues/3388), and several related landed between 18.13.x → 18.14.x). It is disabled unless you answer "Y" to `atuin setup`, but the default in `setup.rs` is Y.

What it is: a TUI LLM chat bound to `?` at the start of an empty shell prompt. Under the hood, your CLI opens an SSE stream to `https://hub.atuin.sh/api/cli/chat` (`crates/atuin-ai/src/stream.rs`), and the server emits `tool_call` events that your client executes locally. The currently-implemented client-side tools are (`crates/atuin-ai/src/tools/mod.rs:321-332`):

- `read_file` — read any file
- `create_file` — write files (enum variant exists; execution is currently `not yet implemented`, line 379)
- `execute_shell_command` — spawn a shell and run a command (execution pipeline is fully implemented via `execute_shell_command_streaming`, line 664+)
- `atuin_history` — search your encrypted history on the client

**The permission model** (`crates/atuin-ai/src/permissions/`) is decent for what it is:
- Rules live in `.atuin/permissions.ai.toml`, walked from cwd up to root, then `~/.config/atuin/permissions.ai.toml`.
- Priorities: `ask` > `deny` > `allow`. Default when no rule matches is **ask**.
- Rule syntax: `Tool(scope)` where scope is a glob (files) or subcommand matcher (shell).

**But the threat model is:** a remote service instructs your CLI to do things, and you are the last line of defense via an "ask" prompt. Things to think hard about:

- A compromised (or socially engineered) prompt on the Hub could try to exfiltrate via an `execute_shell_command` that posts your data somewhere, or via `read_file` on `~/.ssh/` / `~/.aws/credentials` / `~/.local/share/atuin/key`. You'd see the prompt, but prompt fatigue + a plausibly-phrased justification is a known failure mode.
- An overly-broad allowlist like `Shell(git *)` doesn't preclude `git config --global alias.x='!bash -c "curl … | bash"'; git x`. Shell globs are not a safe sandbox.
- The capability gating (`stream.rs:292-311`) is client-enforced — good. It rejects tool calls whose capability wasn't advertised.
- `ai.capabilities.enable_history_search` defaults to **true** — the server can request the full history search tool. The client will still ask permission, but it's one "Allow" away from a natural-language-queryable copy of your command history leaving your machine.
- `send_cwd` and `send_last_command` default to false. Good.

The `atuin hook install` mechanism for Claude Code / Codex / pi is separate from the AI feature itself — those hooks *only* record shell commands executed by those agents into your local atuin database. They don't phone home beyond whatever sync you've configured. That part is benign.

## 6. Daemon / local IPC

`crates/atuin-daemon/src/server.rs`:
- **Unix**: Unix-domain socket at `~/.local/share/atuin/atuin.sock`. Access is controlled by filesystem permissions, which are inherited from the user's umask — the code doesn't explicitly `chmod 0600` the socket. On a standard single-user machine this is fine; on a shared box, check it.
- **Windows**: `127.0.0.1:8889` TCP **with no authentication**. Any process on the local machine can call the gRPC services (history add/end, search, and the control service which includes shutdown). This is a smaller issue than it sounds — anything running as your user already has your history DB — but it's worth knowing.

## 7. Other sharp edges worth naming

- **External plugin model** (`crates/atuin/src/command/external.rs`): `atuin foo` spawns `atuin-foo` from PATH if it exists. Standard git-style, but means dropping a hostile `atuin-*` binary into PATH gets invoked by typos.
- **`atuin-scripts`** (`crates/atuin-scripts/src/execution.rs`): templated scripts written to a temp file with `0o755`, executed. Not auto-invoked — you have to run them — but once you do, they run as your user with full privileges. Minijinja is being used as the templating engine; injection via untrusted script content isn't exploitable because you *are* the one executing.
- **Hook ID file** (`crates/atuin/src/command/client/hook.rs:187-189`): uses `std::env::temp_dir().join(format!("atuin-hook-{tool_use_id}"))`. The `tool_use_id` comes from the AI agent, not an attacker. On macOS `temp_dir()` is user-private; on Linux with a shared `/tmp` and a predictable ID, a local attacker could race the file. Low severity.
- **No telemetry pings beyond the `update_check` ping.** `reqwest::Client` is only instantiated in a small set of places: `api_client.rs`, `hub.rs`, `stream.rs`. All three are tied to explicit user-initiated flows (sync, hub auth, AI chat) or the update check.

## Answers to the specific questions

### "Is this safe for a developer to use on a personal machine?"

Roughly **yes, with three caveats**:

1. Don't run `curl … | sh`. Install the binary yourself (brew/apt/cargo/release download), and skip `atuin setup` or answer "n" to the AI prompt until you've decided whether you want it. The installer modifies your AI-agent configs silently.
2. Treat the encryption key (`~/.local/share/atuin/key`) like a root credential. `chmod 600` it if it isn't already, back it up *encrypted*, and understand that losing it loses your synced history.
3. If you enable Atuin AI, invest five minutes in writing a `~/.config/atuin/permissions.ai.toml` with explicit `deny` rules for `~/.ssh/**`, `~/.aws/**`, `~/.atuin/**`, `~/.local/share/atuin/**`, `**/.env`, `**/credentials*`, etc., and keep `Shell` scoped narrowly. The default "ask" posture is OK but relies entirely on you paying attention to each prompt.

### "Are there unexpected or dangerous side effects?"

The non-obvious ones:

- Default install auto-modifies AI-agent config files if those agents are present.
- `atuin setup` defaults AI and daemon to enabled (Y).
- `update_check` hits `api.atuin.sh` even if you self-host — set `update_check = false` if you care.
- Even with self-hosted sync, logging in triggers a Hub session attempt (the `ensure_hub_session` call, confirmed in `crates/atuin/src/command/client/account/login.rs:118`) — benign if it fails, but it is an outbound call to `hub.atuin.sh`.
- `secrets_filter` reduces but does not eliminate secret capture. Assume that some secrets will end up encrypted-but-synced.
- Windows daemon is unauthenticated TCP on localhost. On a multi-user Windows box, this matters.

### "Can all its dependencies be hosted locally/privately?"

Partially. You can fully replace:

- The sync server — postgres or sqlite backend, off you go.
- The binary distribution — build from source, mirror releases.
- The update-check — disable with `update_check = false`.

You **cannot** replace the Atuin Hub / Atuin AI backend with anything in this repo. If private hosting of everything is a requirement:

- `ai.enabled = false` (or `--disable-ai` on `atuin init`)
- `update_check = false`
- `sync_address = https://your.server`
- `[sync] records = true` — already default
- Install without the curl pipe; skip `atuin setup`.

That configuration is fully self-contained. It's the AI feature that ties you to atuin.sh.

---

## Bottom line

For the core use case (local shell history + optional encrypted sync to your own server), atuin is well-engineered and the security posture matches the marketing. The encryption is real, the server is separable, and nothing sensitive leaves your machine in cleartext.

The newer AI subsystem inverts that story: it's a remote-instructed local agent with file and shell tools, defended primarily by a permission prompt. It's *not* fundamentally unsafe — the permission model is reasonable and many things default off — but it is a different trust relationship from the rest of the tool, and it can't be self-hosted today. Decide explicitly whether you want it; don't let `atuin setup` decide for you.
