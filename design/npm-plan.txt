## Architecture shape (as shipped)

Three orthogonal pieces:

1. **Manifest parsing** — `internal/manifest/npm/`. Returns
   `[]manifest.Dep` identical to gomod's shape; survey consumes
   it without special-case code.
2. **Registry client** — `internal/signal/registry/npm/client.go`.
   HTTPS to `registry.npmjs.org` (main metadata) and `api.npmjs.org`
   (downloads). Models `signal/github/client.go` for
   defense-in-depth.
3. **Signal collector** — `internal/signal/registry/npm/collector.go`.
   Implements `signal.Collector`; scheme-filtered to `pkg:npm/*`.

Orchestration in `analyze.go`:

- `profile.ResolveTarget(cmd.Target)` → canonical URI + scheme
- If `scheme == pkg` + `ecosystem == npm`: call npm client for
  `repository.url`, structurally validate, stamp on entity
- `collectorsFor(entity)` returns the applicable subset based on
  entity type + resolved URL

Client surface:

- `GetPackage(ctx, name) (*RegistryPackage, error)` — full metadata
- `GetWeeklyDownloads(ctx, name) (int, error)` — downloads API
- `NormalizeDeclaredRepoURL(url) (string, bool)` — repo-URL shape
  normalization for the A.5 stamping step

`RegistryPackage` models what the collector reads: `Time`
(per-version publish timestamps), `Maintainers`, `Versions` map
with per-version `Scripts` (postinstall), `Dist.Attestations`
(OIDC), `NpmUser.Name/Email` (publisher identity — added for B.6),
`Repository.URL`. Unused fields unmodeled;
`json.DisallowUnknownFields` surfaces registry drift as a test
failure we'd want to know about.

## Phase A — shipped 2026-04-20 (093a6dd)

Commit: `npm Phase A: provider shell + analyze refactor + first signal end-to-end`

Delivered:

- `analyze.go` adopts `ResolveTarget` and branches entity creation
  on scheme (`repo:*` → EntityProject; `pkg:npm/*` → EntityPackage).
- `collectorsFor` gated by entity type; npm-only analyze doesn't
  require `--path`/`--clone`.
- npm registry client with HTTPS-only redirect, 10MB response cap,
  body-free error strings, `DisallowUnknownFields`, 60s timeout,
  pre-HTTP name validation (allowlist + scoped-name grammar).
- npm collector scheme-filtered to `pkg:npm/*`.
- First signal: `last_publish` (vitality / medium FR).
- A.5 repo resolution: `repository.url` extracted (string or
  `{type,url}` shape), structurally validated, github-hosts
  allowlisted.
- E2E functional test with httptest servers for npm + github.

Tech-debt byproducts (ResolveTarget adoption + collectorsFor
generalization) are reusable for future ecosystem providers.

## Phase B.6 — shipped 2026-04-20 (5f5519b)

Commit: `npm Phase B.6: longitudinal signals (postinstall_introduced, publish_origin_consistency)`

Added mid-stream after a parallel analysis observed that every
Phase A/B signal was single-version-of-latest metadata. An
axios-style attack — one malicious patch, one compromised account —
cannot be distinguished from a legitimate publish using snapshot
data alone. The pattern is in the transition, not the state.

Delivered signals (no additional HTTP traffic — derived from the
already-fetched `RegistryPackage`):

| Signal | Source | Group | Forgery resistance |
|---|---|---|---|
| `postinstall_introduced` | postinstall transition over latest N versions | publication | high |
| `publish_origin_consistency` | `_npmUser` + attestation distribution across versions | publication | very_high |

Implementation notes:

- `PackageVersion` struct gained `NpmUser` (Name, Email). Email
  parsed but not emitted (PII).
- Versions ordered by `pkg.Time[version]` publish timestamp,
  newest-first. Versions without a corresponding time entry are
  skipped rather than ordered by semver fallback — a wrong order
  silently corrupts longitudinal signals, and "can't order" is
  safer than "order wrong."
- Cross-version window bounded at 10 (`crossVersionWindow`).
  Tested with a 14-version fixture to confirm the cap blocks
  false transition detection when an old-version postinstall
  sits outside the window.
- Both signals record absence (not bogus value) when the version
  set is unorderable — consistent with the "crashing over data
  corruption" rule.

The publisher-identity data B.6 started collecting is the
signal-side groundwork for the per-publisher trust profile named
in `analysis-economics.md` §8 and `analyst-composability.md` §1.
Not a full amortization system yet — just the raw material.

Live dogfood (`pkg:npm/zod`): produced the "solo-but-coherent"
pattern the signals were designed to distinguish from an
axios-style transition (single publisher, consistent attestation
presence, zero transitions).

## Phase C — shipped 2026-04-20 (f203226)

Commit: `npm Phase C: manifest parser + survey integration`

Delivered:

- `internal/manifest/npm/parse.go` parses `package.json` (required)
  and `package-lock.json` v3 (optional, for transitive resolution).
- Four dep classes handled: `dependencies`, `devDependencies`,
  `peerDependencies`, `optionalDependencies`. Direct entries
  `Direct=true`; transitives from the lockfile `Direct=false`.

  scoped.

  `Ecosystem="npm-local"`, empty CanonicalURI (mirrors gomod
  `go-local-replace` pattern).

  skill's npm claim now cashed end-to-end.
- testdata fixtures: `simple/`, `scoped/`, `mixed/`, `no-lockfile/`.

## Security invariants (enforced across all npm code)

- `ValidateCanonicalURI` at persistence boundary
- HTTPS-only redirect in npm client (#89)
- Max response size cap (10MB per github-client pattern)
- Error sanitization: response body NEVER in error string (#93)
- Package-name validation before URL construction (allowlist +
  scoped-name grammar)
- Context cancellation propagation tested (#87 pattern)
- `json.DisallowUnknownFields()` on every registry-response struct
- `signalTypeRegistry` entry for every new signal type (otherwise
  `signal.Make` panics at first emission)

## Test discipline (observed)

- Table-driven unit tests with explicit `name` fields and
  `t.Run(tc.name, …)` per case
- TDD for security-adjacent fixes: failing test before the fix
- Writer injection (per survey's pattern) for any stdout output;
  no global `os.Stdout` redirection — breaks `t.Parallel()`
- Functional tests via `httptest`, never live npm/github

## Out of scope (still post-v0.1)

- yarn.lock / pnpm-lock.yaml
- Git tag correspondence provenance signal
- Tarball-vs-source content diff
- Authenticated npm registry requests (tokens)
- Private/scoped registries
- Cross-ecosystem correlation
- TTL cache on registry responses (ROADMAP should-do)

## Next

Pending npm work, ordered roughly by readiness.

1. **Composition group — `direct_dep_count` then
   `transitive_dep_count`.** New signal group for measurements of
   the target's composed surface (vs. evidence about its
   publisher). `direct_dep_count` is zero incremental cost — the
   `dependencies` map is already in the fetched `RegistryPackage`.
   `transitive_dep_count` requires either a tree-walk (N HTTP
   calls) or a parsed lockfile; the lockfile path is preferred
   since Phase C already parses it. ForgeryResistance does not
   apply (composition signals describe the object, not claims
   someone made about it). Contract: surface the shape, do not
   produce a verdict — org-policy decides acceptability. See
   `analyst-composability.md` §6 for the skim-mode motivation.

2. **Derived signals over the B.6 longitudinal data.** Phase B.6
   collects publisher distribution per-version but does not yet
   produce derivatives like `has_historically_published_with_oidc`
   or `maintainer_account_age_spread`. Cheap to add once dogfood
   shows which derivatives are decision-relevant.

3. **Per-publisher profile (stretch).** The `_npmUser` data B.6
   started collecting points toward the amortization model named
   in `analysis-economics.md` §8 and `analyst-composability.md`
   §1. Needs a store-side shape (publisher table keyed on
   `(ecosystem, maintainer_id)`) before a cross-package profile
   is meaningful. Not scoped yet — wants 5-10 dogfood runs from
   the same maintainer first.

## Open questions

1. **`dependent_count` vs. `direct_dep_count`:** Phase B dropped
   `dependent_count` (gameable inbound-popularity signal). Fill
   the empty slot with `direct_dep_count` in the new `composition`
   group — same external-traffic cost (zero additional calls; the
   `dependencies` map is already in the fetched struct), stronger
   signal, directly supports skim-mode triage of high-fanout
   targets. Default position: add `direct_dep_count`; do not
   reinstate `dependent_count`.

## Closed questions

- **OQ2 Survey tie-breaking** (from the pre-implementation plan):
  if both `go.mod` and `package.json` live in the same directory,
  first-found wins — OK, or do we want explicit ordering? Closed:
  `manifest.Detect` iterates a fixed-order candidate list
  (`go.mod`, `package.json`) and returns the first existing file;
  the `--manifest` flag is the escape hatch for multi-manifest
  polyglot roots. Behavior documented inline in
  `internal/manifest/detect.go`.
