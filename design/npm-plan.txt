  The question splits into two distinct interpretations, and the answer is different for each.

  Interpretation A — Gateway verification (does the URL even resolve?)

  When the npm registry says repository.url: "git+https://github.com/expressjs/express.git", do we proactively check that the
  GitHub repo exists before using it?

  My recommendation: don't verify proactively. Use the registry's claim as-is and let downstream collection fail gracefully if
  it's bad.

  Reasons:
  - An extra API call per package, just to check existence, doubles the network cost for what is almost always a successful
  check
  - The github collector already has a fail-graceful path: if the resolved repo URL doesn't exist, client.GetRepo returns "not
  found" and the collector records absence signals for the github-side tier. The user/LLM sees "we couldn't get any github
  signals for this entity" — which is the right surface for "the npm package's repo claim was wrong or stale"
  - The threat model isn't "an attacker tricked us into making one bogus API call"; it's "an attacker convinced us a package is
  trustworthy when it isn't"

  So for the lookup path: trust the registry, structural validate the URL with the existing validateContentsPath-style allowlist
   (so it at least looks like a real GitHub URL and not javascript: or file://), then hand it off downstream. Bad URLs fail
  naturally during signal collection.

  Interpretation B — Provenance verification (did this published version actually come from that repo?)

  This is the substantive question — the one the axios case study explicitly identifies as a high-value signal class. Three real
   verification mechanisms exist:

  1. OIDC trusted publishing attestation. The npm registry exposes an attestations field (when present) for packages published
  via OIDC trusted publishing. The attestation is cryptographically signed by the npm registry and binds the published version
  to a source repo + commit SHA. This is the gold standard — no extra API calls, cryptographically verifiable, and per axios
  lesson #13 "Whether a package uses OIDC-based trusted publishing is a binary, queryable signal." When present and valid, it's
  a strong positive signal. When absent on a package that previously used it, that's a strong negative signal (the documented
  axios attack pattern).
  2. Git tag correspondence. For each published npm version, look for a matching git tag in the resolved repo (try v1.2.3,
  1.2.3, release-1.2.3 patterns). Per axios lesson #10: "the malicious axios versions were published without trusted publisher
  binding and without corresponding repo tags or commit trails." This is a HIGH-VALUE signal that directly addresses the
  documented attack pattern. But it requires API calls to BOTH npm AND github for the same entity, plus comparison logic.
  3. Tarball-vs-source content diff. Download the published tarball, compare against the source tree at the corresponding tag.
  Catches "axios source code untouched but package.json modified" attacks. Highest fidelity, highest cost (tarball download per
  version).

  My recommendation for the v0.1 npm collector:
  - Ship #1 (OIDC trusted publishing presence) as a flat metadata signal. It's a single field in the registry response for the
  latest version. No cross-collector orchestration needed. Directly addresses the documented attack pattern. Fits "flat metadata
   only" from Q5.
  - Defer #2 (git tag correspondence) to a follow-up signal. It requires cross-collector orchestration (npm collector reading
  data the github collector also reads), which is a real architectural decision worth its own design pass — does the
  orchestration live in the npm collector, the github collector, or some new comparison-pass? Not the right call to make under
  time pressure on the v0.1 first ship.
  - Defer #3 (tarball diff) to v0.2+. Significant scope (tarball download, content comparison, signed-vs-unsigned semantics).

  So the v0.1 verification story is: trust the URL for lookup, collect OIDC attestation presence as a provenance signal,
  document the deferred verification mechanisms in the commit message. When tag correspondence ships later, it becomes a second
  provenance signal that strengthens the picture.

  This keeps Q3's answer scope-consistent with Q5's "flat metadata only" — OIDC presence is one field in one response. Tag
  correspondence and tarball diffing are separate design exercises that can land iteratively without needing to be perfect on
  day one.

  Phase 4 — analyze.go refactor for multi-collector dispatch

  File: cmd/signatory/analyze.go

  Current behavior (post-#101): takes a target string, runs NormalizeGitHubRepoInput to convert to a github canonical URI, calls
   collectors. Only the github collector exists today, so the dispatch is implicit.

  New behavior:
  - Detect input shape:
    - If NormalizeGitHubRepoInput succeeds → github canonical URI (existing path)
    - Else if ValidateCanonicalURI succeeds → use as-is (npm canonical URIs come in this way: pkg:npm/express)
    - Else → error with helpful message listing accepted forms
  - For each registered collector that "applies" to the entity, call it
  - Merge signal results

  The "applies" question: does the github collector run on a pkg:npm/express entity? Probably yes, IF the npm provider has
  resolved a github repo URL for it (via ResolveRepo). The flow becomes:
  1. Parse target → canonical URI + entity type
  2. If type is package and URI is pkg:npm/...: ask npm provider to resolve to a repo URL
  3. Look up or create the entity, including the resolved repo URL
  4. Run all applicable collectors on the entity

  This is a real refactor — maybe 50-100 lines changed in analyze.go plus a few helper additions. The test coverage is via Phase
   5's end-to-end test, plus existing analyze tests staying green.

  Subtle decision: where does the repo resolution happen? Two options:
  - (a) Inside analyze.go's flow — call the provider explicitly before running collectors
  - (b) Inside the npm Collector itself — when npm collects, it ALSO resolves the repo and updates the entity

  (a) is cleaner architecturally — providers and collectors have different jobs and shouldn't bleed into each other. (b) is more
   cohesive but couples npm collector to entity mutation.

  Recommendation: option (a). Add a step "resolve dependencies' source repos via the appropriate provider" between entity
  creation and signal collection. The provider produces the repo URL; analyze.go updates the entity with it; collectors pick up
  the URL when they need it (the github collector reads entity.URL).
